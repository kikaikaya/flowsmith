# flowsmith 実行エンジン (v1)

フロー定義 YAML を受け取り、宣言されたステップを順に実行する。
**エンジンは推測で進まない。** 解釈に迷う状況はすべて `seized`（停止・人間待ち）として報告する。

## v1 の実装範囲

- ステップタイプ: `make` のみ（`make-check` / `branch` / `loop` を検出したら seized で停止し「未実装」と報告する）
- ゲート: `exists` / `sections` のみ（他のゲートは宣言を無視せず「未実装ゲートあり」と警告する）
- 状態ファイルによる再開: 未実装（毎回先頭から実行する。Phase 2 で導入予定）

---

## Phase 0: 初期化と検証

1. 引数の YAML ファイルを Read で読み込む。
2. `paths` の変数を展開する：
   - `OUT` は**予約変数・必須**（成果物と実行時ファイルの置き場）。無ければ seized。
   - 相対パスは**カレントワーキングディレクトリ**から解決する。以後すべて絶対パスで扱う。
3. 検証（ひとつでも失敗 → seized 報告して終了）：
   - `flow.id` と `steps[]` が存在する
   - 全 step の `id` が一意である
   - 全 step の `type` が既知である（v1 では `make` 以外は「未実装」として seized）
   - `next` の参照先 step id が実在する
   - 各 step に `brief` か `brief_file` のどちらかがある（`brief_file` は実在確認）
4. `${OUT}` ディレクトリを作成する（存在しなければ）。
5. 実行計画を 1 行ずつユーザーに提示する：`[順序] step_id — title`。

## Phase 1: ステップ実行ループ

現在ステップ = steps 配列の先頭から開始。以下を繰り返す：

### 1.1 進捗表示

`[完了数/総数] {step_id}: {title}` を表示する。

### 1.2 派発準備

- `brief_file` 指定なら Read で読み込み、brief として扱う
- `refs` / `inputs` / `outputs` のパスを絶対パスに展開する

### 1.3 Agent ツールで派発

下記の派発プロンプト雛形を埋めて、Agent ツール（general-purpose）で subagent を起動する。

### 1.4 結果ブロックの解析

subagent の最終メッセージから ` ```flowsmith-result ` フェンスを抽出する。

- **ブロックが無い／YAML として不正／`step` が現在 step と不一致** → 同一指示に「結果ブロック欠落。形式を厳守せよ」と添えて **1 回だけ再派発**。再失敗 → seized。
- `status: seized` → そのまま seized 処理へ（subagent の notes を引用する）。
- `status: done` → 次へ。

### 1.5 ゲート評価（エンジン自身が決定的に判定する。LLM に聞かない）

各 gate を順に検査：

| gate | 判定方法 |
|------|----------|
| `exists: <path>` | ファイルが実在する（Glob/Read で確認） |
| `sections: [...]` | 出力ファイル内に各見出し文字列が存在する（Grep で確認） |

- 不合格 gate があれば、その一覧を添えて **1 回だけ再派発**（「以下のゲートを満たしていない。成果物を修正せよ」）。再不合格 → seized。
- v1 未実装の gate（`min_lines` 等）は評価せず、完了報告で「未評価ゲート」として明示する。

### 1.6 ルーティング

- `next` があればその step へ。無ければ配列の次の step へ。
- 末尾 step の完了 → Phase 2（完了報告）へ。

## Phase 2: 完了報告

ユーザーに提示する：

1. 実行した step の一覧（id / title / 再派発の有無）
2. 成果物のパス一覧
3. 未評価ゲート・警告があればその一覧

## seized 時の報告（フォーマット厳守）

```
⛔ flowsmith seized — 人間の裁定が必要です
- フロー: {flow.id}
- ステップ: {step_id}（{title}）
- 原因: {検証失敗 / 結果ブロック不正 / ゲート不合格 ×2 / 未実装タイプ など具体的に}
- 経緯: {再派発したか、subagent の notes 等}
- 復旧の選択肢: {例: brief を修正して再実行 / ゲートを緩める / 手動で成果物を直す}
```

報告後は**何もせず停止**する。勝手にリトライ・回避をしない。

## 派発プロンプト雛形

```
あなたは flowsmith フローの 1 ステップを実行する作業者です。担当範囲外のことはしないでください。

# ステップ: {title}（{step_id}）

## 作業指示
{brief}

## 参照資料（作業前に Read で読むこと）
{refs を箇条書き。無ければ「なし」}

## 入力
{inputs を「- {id}: {絶対パス} — {note}」形式で}

## 出力（必ずこの絶対パスに書くこと。これ以外のファイルを変更しない）
{outputs を「- {id}: {絶対パス}」形式で}

## 完了時の約束（厳守）
最終メッセージの末尾に、次の形式のフェンスブロックを必ず含めること：

```flowsmith-result
step: {step_id}
status: done
outputs:
  - {実際に書いたパス}
notes: {一行メモ}
```

完遂できない場合も省略せず、status: seized とし notes に理由を書くこと。
```
